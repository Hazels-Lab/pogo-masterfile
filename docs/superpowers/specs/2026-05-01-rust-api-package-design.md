# Rust API Package — Design Spec

**Date:** 2026-05-01
**Author:** brainstorming session (derick.magnusen@gmail.com + Claude)
**Status:** Approved, pending implementation

## Problem

The existing Rust types crate (`pogo-masterfile-types`, at `packages/rust/`) ships generated structs and a `parse_masterfile` function that returns `Vec<MasterfileEntry>`. Consumers wanting to *use* the masterfile still have to:

1. Source `masterfile.json` themselves (or fetch it from a community mirror).
2. Build their own indexes (`HashMap<TemplateId, &Entry>`, per-group buckets) every time they want O(1) lookups.
3. Reinvent error handling around the network + parse stages.

This mirrors exactly the gap the TypeScript API package (`pogo-masterfile`, at `packages/ts-api/`) closed for JavaScript consumers. This spec covers the Rust equivalent.

## Goal

Ship a sibling crate, `pogo-masterfile`, that depends on `pogo-masterfile-types` and provides a runtime `Masterfile` struct with:

- Async loading from the canonical community mirror (with overridable URL and fetcher), via a builder.
- O(1) lookups by templateId.
- Per-group accessors (`mf.move_settings().get(id)`) where the input ID can be either a typed `MoveSettingsTemplateId` enum variant (compile-time validated) or `&str` (runtime parsed).
- Iteration, search, and refresh primitives.
- An optional blocking variant (`pogo_masterfile::blocking::Masterfile`) under a feature flag, for sync consumers.

Like the existing language packages, every file under `packages/rust-api/src/` is owned by the codegen — consumers never edit them, contributors never edit them, only the generator does.

## Decisions & rationale

Captured from the brainstorming session, in order:

1. **Async story: dual feature flags, async default.** `default = ["async"]`, opt-in `blocking`. Both APIs live in the same crate (mirroring reqwest's pattern). Async lives at the crate root; blocking lives in a `blocking` submodule, same struct names.
2. **HTTP client: reqwest.** Supports both async + blocking via its own features. Configured `default-features = false, features = ["rustls-tls", "json"]` to skip openssl baggage.
3. **Polymorphic ID input via `TryInto`.** A single `get<I: TryInto<MoveSettingsTemplateId>>(id: I)` accepts both the typed enum variant AND `&str` (which goes through `FromStr`). Returns `None` on either invalid string OR missing entry.
4. **Borrowed reads, `&mut self` refresh.** Most idiomatic Rust pattern. Zero-copy reads. The compiler statically prevents concurrent reads during refresh — refresh requires unique access. Consumers needing concurrent access wrap `Masterfile` in `Arc<RwLock<_>>` themselves.
5. **Per-group accessor mechanism: method per group.** `mf.move_settings()` returns `MoveSettingsAccessor<'_>` borrowing from `mf`. ~60 generated methods on `Masterfile`. Storage stays as one `Vec<MasterfileEntry>` with per-group index maps; accessors hold borrows. Idiomatic, zero-copy throughout.
6. **Single `Error` enum.** thiserror-derived; one variant per failure mode (`Fetch`, `Status`, `Parse`, `InvalidShape`, `Custom`). The `Custom` variant lets user fetchers wrap any error type behind a `Box<dyn Error + Send + Sync>`.

**Crate name: `pogo-masterfile`** — matches the npm package name. The pairing is `pogo-masterfile` (runtime) + `pogo-masterfile-types` (types-only), exactly mirroring the TS layout.

**Workspace registration:** root `Cargo.toml` adds `packages/rust-api` to its `members` array. The crate depends on `pogo-masterfile-types = { path = "../rust", version = "0.1" }` (workspace path during dev, version-pinned at publish).

**Cargo features:**

```toml
[features]
default = ["async"]
async = ["dep:tokio", "dep:reqwest", "reqwest/rustls-tls", "reqwest/json"]
blocking = ["dep:reqwest", "reqwest/blocking", "reqwest/rustls-tls", "reqwest/json"]
```

`async` brings tokio + reqwest's async client. `blocking` brings reqwest's blocking client. Both can be enabled simultaneously.

**No build artifacts to commit.** Unlike ts-api's `dist/`, Rust crates ship source — `cargo` builds on the consumer's machine. The codegen pipeline writes only `src/` and runs `cargo fmt`. No `cargo` invocation beyond formatting.

**Tests / examples directories are hand-written**, sibling to `src/`, NOT regenerated by codegen — same pattern as `packages/ts-api/tests/`.

### Public API surface (async, crate-root)

```rust
// pogo_masterfile/src/lib.rs

pub use crate::error::{Error, Result};
pub use crate::fetcher::{Fetcher, ReqwestFetcher, DEFAULT_MASTERFILE_URL};
pub use crate::masterfile::Masterfile;
pub use crate::builder::MasterfileBuilder;

// Re-export upstream so consumers don't need both crates in their imports.
pub use pogo_masterfile_types::{MasterfileEntry, parse_masterfile};

// Per-group accessor types — one per group, generated.
pub mod accessor;

#[cfg(feature = "blocking")]
pub mod blocking;
```

```rust
// Builder pattern for from_remote — idiomatic for "constructor with optional config".
pub struct MasterfileBuilder { /* private fields */ }

impl MasterfileBuilder {
    pub fn url(self, url: impl Into<String>) -> Self;
    pub fn timeout(self, timeout: Duration) -> Self;
    pub fn fetcher(self, fetcher: impl Fetcher + 'static) -> Self;

    /// Async terminal call.
    pub async fn fetch(self) -> Result<Masterfile>;
}

// Main type.
pub struct Masterfile {
    pub(crate) entries: Vec<MasterfileEntry>,
    pub(crate) by_id: HashMap<String, usize>,
    pub(crate) groups: GroupIndexes, // generated struct holding all per-group HashMaps + order vecs
}

impl Masterfile {
    // ── Loaders ──────────────────────────────────────────────────
    pub fn builder() -> MasterfileBuilder;

    /// Convenience: `Masterfile::builder().fetch().await`.
    pub async fn from_remote() -> Result<Self>;

    /// Parse from a JSON string (sync — no I/O).
    pub fn from_json(json: &str) -> Result<Self>;

    /// Wrap an already-parsed entry vec.
    pub fn from_entries(entries: Vec<MasterfileEntry>) -> Self;

    // ── Top-level lookups ────────────────────────────────────────
    pub fn get_entry(&self, id: &str) -> Option<&MasterfileEntry>;
    pub fn has(&self, id: &str) -> bool;

    // ── Iteration / introspection ────────────────────────────────
    pub fn entries(&self) -> &[MasterfileEntry];
    pub fn len(&self) -> usize;
    pub fn is_empty(&self) -> bool;
    pub fn iter(&self) -> std::slice::Iter<'_, MasterfileEntry>;
    pub fn template_ids(&self) -> impl Iterator<Item = &str> + '_;

    // ── Per-group accessors (~60 generated methods) ──────────────
    pub fn move_settings(&self) -> accessor::MoveSettingsAccessor<'_>;
    pub fn pokemon_settings(&self) -> accessor::PokemonSettingsAccessor<'_>;
    // ... ~60 methods, each a one-liner returning a borrow struct ...
    pub fn singletons(&self) -> accessor::SingletonsAccessor<'_>;

    // ── Mutation (require &mut self — invalidates outstanding borrows) ──
    pub async fn refresh(&mut self) -> Result<()>;
    pub fn update(&mut self, entries: Vec<MasterfileEntry>);
}

// IntoIterator so `for entry in &mf { ... }` works.
impl<'a> IntoIterator for &'a Masterfile {
    type Item = &'a MasterfileEntry;
    type IntoIter = std::slice::Iter<'a, MasterfileEntry>;
    fn into_iter(self) -> Self::IntoIter { self.iter() }
}
```

### Per-group accessor (generated, one per group)

```rust
// pogo_masterfile/src/accessor/move_settings.rs
use std::collections::HashMap;
use pogo_masterfile_types::{MasterfileEntry, MoveSettingsEntry, MoveSettingsTemplateId};

pub struct MoveSettingsAccessor<'a> {
    pub(crate) entries: &'a [MasterfileEntry],
    pub(crate) index: &'a HashMap<MoveSettingsTemplateId, usize>,
    pub(crate) order: &'a [usize],
}

impl<'a> MoveSettingsAccessor<'a> {
    /// Polymorphic input: accepts `MoveSettingsTemplateId` (compile-checked)
    /// or `&str` (runtime-parsed via `FromStr`).
    pub fn get<I>(&self, id: I) -> Option<&'a MoveSettingsEntry>
    where
        I: TryInto<MoveSettingsTemplateId>,
    {
        let typed = id.try_into().ok()?;
        let idx = *self.index.get(&typed)?;
        match &self.entries[idx] {
            MasterfileEntry::MoveSettings(e) => Some(e),
            _ => None, // generator invariant: index points at the right variant
        }
    }

    pub fn has<I: TryInto<MoveSettingsTemplateId>>(&self, id: I) -> bool {
        self.get(id).is_some()
    }

    pub fn iter(&self) -> impl Iterator<Item = &'a MoveSettingsEntry> + '_ {
        self.order.iter().filter_map(|&idx| match &self.entries[idx] {
            MasterfileEntry::MoveSettings(e) => Some(e),
            _ => None,
        })
    }

    pub fn template_ids(&self) -> impl Iterator<Item = MoveSettingsTemplateId> + '_ {
        self.index.keys().copied()
    }

    pub fn len(&self) -> usize { self.order.len() }
    pub fn is_empty(&self) -> bool { self.order.is_empty() }
}
```

Note: no `IntoIterator` impl on the accessor. Users iterate explicitly via `.iter()`. The accessor's iter is a `filter_map` chain (not a slice), so naming the iterator type cleanly would require either `impl Trait` aliases (workable on Rust 1.85, but adds noise) or boxing. `.iter()` is enough.

### Fetcher trait + default impl

```rust
// pogo_masterfile/src/fetcher.rs

pub const DEFAULT_MASTERFILE_URL: &str =
    "https://raw.githubusercontent.com/alexelgt/game_masters/refs/heads/master/GAME_MASTER.json";

/// User-replaceable fetcher. Implement on your own struct, OR pass a closure
/// — closures matching the right shape get a free impl below.
pub trait Fetcher: Send + Sync {
    async fn fetch(&self, url: &str) -> Result<Vec<MasterfileEntry>>;
}

impl<F, Fut> Fetcher for F
where
    F: Fn(&str) -> Fut + Send + Sync,
    Fut: std::future::Future<Output = Result<Vec<MasterfileEntry>>> + Send,
{
    async fn fetch(&self, url: &str) -> Result<Vec<MasterfileEntry>> {
        (self)(url).await
    }
}

/// Default: reqwest with rustls + JSON parse + array-shape validation.
#[cfg(feature = "async")]
pub struct ReqwestFetcher {
    client: reqwest::Client,
}

#[cfg(feature = "async")]
impl ReqwestFetcher {
    pub fn new() -> Self;
    pub fn with_client(client: reqwest::Client) -> Self;
}

#[cfg(feature = "async")]
impl Fetcher for ReqwestFetcher {
    async fn fetch(&self, url: &str) -> Result<Vec<MasterfileEntry>> {
        let response = self.client.get(url).send().await
            .map_err(|source| Error::Fetch { url: url.into(), source })?;
        if !response.status().is_success() {
            return Err(Error::Status {
                url: url.into(),
                status: response.status().as_u16(),
            });
        }
        let body = response.text().await
            .map_err(|source| Error::Fetch { url: url.into(), source })?;
        parse_masterfile(&body).map_err(Error::from)
    }
}
```

The closure auto-impl uses Rust 1.75+ `async fn in trait` (workspace already pins `rust-version = "1.85"`). No `async-trait` macro dependency.

### Error type

```rust
// pogo_masterfile/src/error.rs
use thiserror::Error;

#[derive(Debug, Error)]
pub enum Error {
    #[error("network error fetching {url}")]
    Fetch {
        url: String,
        #[source]
        source: reqwest::Error,
    },

    #[error("HTTP {status} from {url}")]
    Status { url: String, status: u16 },

    #[error("failed to parse masterfile JSON")]
    Parse(#[from] serde_json::Error),

    #[error("expected JSON array of entries, got something else")]
    InvalidShape,

    #[error("custom fetcher error")]
    Custom(#[source] Box<dyn std::error::Error + Send + Sync>),
}

pub type Result<T> = std::result::Result<T, Error>;
```

### Blocking submodule

```rust
// pogo_masterfile/src/blocking.rs (template + generated per-group methods)
//
// Same struct fields and storage as crate-root Masterfile. Differences:
// - `from_remote()` is sync (uses reqwest::blocking::Client)
// - `refresh(&mut self)` is sync
// - Builder's terminal call is `.fetch()` (no .await)
//
// The per-group accessor methods and types are SHARED with the async path —
// `pogo_masterfile::accessor::MoveSettingsAccessor<'_>` is the same struct
// either way. No async appears in lookups, so there's nothing to fork.

#[cfg(feature = "blocking")]
pub struct Masterfile { /* same fields as crate root */ }

#[cfg(feature = "blocking")]
impl Masterfile {
    pub fn builder() -> MasterfileBuilder;
    pub fn from_remote() -> Result<Self>;
    pub fn from_json(json: &str) -> Result<Self>;
    pub fn from_entries(entries: Vec<MasterfileEntry>) -> Self;

    // ... same lookup / iteration methods ...

    pub fn refresh(&mut self) -> Result<()>;
    pub fn update(&mut self, entries: Vec<MasterfileEntry>);

    // Same ~60 generated per-group accessor methods.
}
```

The blocking variant has its own `MasterfileBuilder` (different terminal call signature). Per-group accessor structs are SHARED — `pogo_masterfile::accessor::MoveSettingsAccessor<'_>` works identically with either struct, since lookups have no async surface.

### Codegen pipeline integration

The orchestrator `src/generate.ts` adds one more generator:

```ts
await Promise.all([
  (async () => {
    await generateTypeScript(entries);
    await generateTypeScriptApi(entries);
  })(),
  (async () => {
    await generateRust(entries);
    await generateRustApi(entries);          // ← NEW
  })(),
  generateGo(entries),
]);
```

`rust-api` runs after `rust` because its emitted code depends on the upstream type names being current.

**Generator emission map:**

| Output file | Source | Content |
|---|---|---|
| `Cargo.toml` | `emit-cargo.ts` | Package metadata, deps, features. Generated to keep version pinning + workspace path consistent. |
| `lib.rs` | `emit-lib.ts` | `pub use` re-exports + the `impl Masterfile { … }` block containing the ~60 per-group accessor methods + the `GroupIndexes` struct that holds all per-group HashMaps and order vecs. |
| `masterfile.rs` | template-copied | Masterfile struct definition + all top-level methods. The template references `GroupIndexes` (defined in lib.rs) so it's generic over the group set. |
| `error.rs` | template-copied | Error enum + Result. |
| `fetcher.rs` | template-copied | Fetcher trait + ReqwestFetcher. |
| `builder.rs` | template-copied | MasterfileBuilder. |
| `blocking.rs` | template-copied + codegen append | Same shape as masterfile.rs but sync. The per-group accessor methods are generated and appended (same shape as the async impl). |
| `accessor/mod.rs` | `emit-accessor-mod.ts` | `pub mod move_settings;` per group + `pub use` re-exports. |
| `accessor/<group>.rs` | `emit-accessor.ts` | `<Group>Accessor<'_>` struct + impl, ~50 lines each. |

**`cargo fmt` runs after each codegen pass,** matching how the existing rust generator's output is formatted. The root `package.json` script gets extended:

```json
"generate": "bun run src/generate.ts && bun run format && (cd packages/rust && cargo fmt) && (cd packages/rust-api && cargo fmt) && (cd packages/go && gofmt -w .)"
```

**No `cargo build` invocation during codegen.** Rust crates ship source — consumers compile on install. Validation happens via `cd packages/rust-api && cargo check` (run separately or on CI).

### Storage layout

```rust
struct Masterfile {
    entries: Vec<MasterfileEntry>,                      // single source of truth
    by_id: HashMap<String, usize>,                      // global lookup → entries index
    groups: GroupIndexes,                               // generated container
}

// Generated by emit-lib.ts (lives in lib.rs alongside the impl block):
struct GroupIndexes {
    move_settings_index: HashMap<MoveSettingsTemplateId, usize>,
    move_settings_order: Vec<usize>,
    pokemon_settings_index: HashMap<PokemonSettingsTemplateId, usize>,
    pokemon_settings_order: Vec<usize>,
    // ... per group ...
    singletons_index: HashMap<SingletonsTemplateId, usize>,
    singletons_order: Vec<usize>,
}

impl GroupIndexes {
    fn build(entries: &[MasterfileEntry]) -> Self {
        // Walk entries once, dispatch by variant into the right index + order vec.
    }
}
```

Indices and order vecs hold `usize` indices into the global `entries` Vec. Per-group accessor borrows are `&self.entries`, `&self.groups.move_settings_index`, `&self.groups.move_settings_order`. All zero-copy.

### Testing strategy

Five layers, each with a clear audience.

**1. Generator unit tests** (`src/rust-api/*.test.ts`, run via `bun test`). Mirror the TS API generator tests:

- `emit-accessor.test.ts` — feed a fixture group, assert struct/impl shape.
- `emit-lib.test.ts` — assert ~60 per-group accessor methods emitted in alphabetical order.
- `emit-cargo.test.ts` — assert features section, dep versions, package metadata.
- `emit-accessor-mod.test.ts` — assert barrel re-exports each group.

Reuse `MOCK_MASTERFILE` from `src/fixtures.ts`.

**2. Crate integration tests** (`packages/rust-api/tests/*.rs`, run via `cargo test`). Mirror the ts-api example tests — usage demos that also verify behavior:

- `tests/fixture.rs` — shared `fixture()` returning a small `Vec<MasterfileEntry>` with real entries from a few groups + a singleton. Plus `mock_fetcher(entries)` helper.
- `tests/loading.rs` — `from_json`, `from_entries`, `Masterfile::builder().fetch().await` with mock fetcher, default URL verification, error paths.
- `tests/lookups.rs` — top-level `get_entry` returns `&MasterfileEntry`, accessor `get(typed_id)` returns `&MoveSettingsEntry`, accessor `get("string")` works, polymorphic `TryInto` confirmed.
- `tests/iteration.rs` — `for entry in &mf`, `mf.iter().filter(...)`, accessor `iter()`, `len`/`is_empty`.
- `tests/mutation.rs` — `update()` swaps data, `refresh()` re-fetches via injected mock fetcher, `&mut self` borrow rules.
- `tests/errors.rs` — `match err { Error::Fetch { .. } => …, Error::Parse(..) => … }`, custom variant.
- `tests/blocking.rs` — gated `#[cfg(feature = "blocking")]`, mirrors loading + lookups for the sync API.

Mocking the fetcher is trivial because `Fetcher` is a trait — tests inject a stub:

```rust
struct MockFetcher(Vec<MasterfileEntry>);
impl Fetcher for MockFetcher {
    async fn fetch(&self, _: &str) -> Result<Vec<MasterfileEntry>> { Ok(self.0.clone()) }
}
```

No `wiremock` or `mockito` dep needed.

**3. Doctests** — embedded in `src/lib.rs` and key types. Run as part of `cargo test --doc`. A handful on the `Masterfile` struct, the accessor pattern, and the builder. The template files include them; codegen preserves them verbatim during the copy step.

**4. Examples** (`packages/rust-api/examples/*.rs`, runnable via `cargo run --example NAME`). Lightweight showcase code, deliberately not test-asserted — for vibe-checks (the Rust equivalent of the ts-api `tests/` showcase):

- `examples/load_async.rs` — full async lifecycle.
- `examples/load_blocking.rs` — same in blocking mode (gated `#[cfg(feature = "blocking")]`).
- `examples/custom_fetcher.rs` — implementing `Fetcher` for a custom struct AND for a closure.
- `examples/per_group.rs` — accessor narrowing demos.

**5. E2E smoke** (`packages/rust-api/tests/e2e.rs`, gated). `#[ignore]`'d so default `cargo test` skips it; run via `cargo test -- --include-ignored`:

```rust
#[tokio::test]
#[ignore = "hits the network — run with --include-ignored"]
async fn e2e_loads_real_masterfile() {
    let mf = Masterfile::from_remote().await.unwrap();
    assert!(mf.len() > 0);
    assert!(mf.move_settings().len() > 100);
    assert!(mf.move_settings().get("V0022_MOVE_MEGAHORN").is_some());
}
```

CI gets one extra step: `cargo test -p pogo-masterfile`.

## Out of scope

- Go API package (`packages/go-api/`) — separate spec, follows the same shape where Go idioms allow.
- Async runtime portability beyond Tokio — if `smol` / `async-std` demand surfaces, that's a follow-up. For v0.1, async path requires Tokio.
- A `cargo` subcommand or CLI bundled with the crate — library-only.
- Schema validation of fetched entries beyond JSON-array shape — same rationale as the TS version: the masterfile is too large for cheap runtime structural validation.
- Caching / persistence layer — out of scope.
- Renaming `packages/rust/` to `packages/rs/` — orthogonal cosmetic change tracked separately.
